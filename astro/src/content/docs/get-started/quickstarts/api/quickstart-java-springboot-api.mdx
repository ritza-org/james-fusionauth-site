This quickstart secures a Spring Boot API resource server with FusionAuth using a preconfigured, local version of FusionAuth in the following steps:

1. Run FusionAuth with Docker.
1. Create a new Spring Boot application.
1. Add JWT-based authentication, model objects, and controllers for two protected endpoints.
1. Get a token and call the protected endpoints.

You'll be building the API for ChangeBank, a bank that converts dollars into coins. It has two endpoints:

* `make-change`: Call this with a `total` amount to get back the number of nickels and pennies needed to make change. The `customer` and `teller` roles can both call this endpoint.
* `panic`: Tellers call this endpoint to alert the police in case of an incident. Only the `teller` role can call this endpoint.

Both endpoints require a valid JSON Web Token (JWT) in the `Authorization` header or in a cookie named `app.at`, and the JWT's `roles` claim must contain the role required by the endpoint.

## Prerequisites

This Quickstart requires:

* [Java 17](https://www.oracle.com/java/technologies/downloads/) — this quickstart's `pom.xml` is pinned to Spring Boot 4.0.7, which requires Java 17. Other compatible Java versions may work but haven't been tested.
* [Maven](https://maven.apache.org/) — the example repository includes a Maven wrapper (`mvnw`), so you don't need to install Maven yourself.
* [Docker](https://docs.docker.com/get-started/get-docker/), with [Docker Compose](https://docs.docker.com/compose/)

## Install and Run FusionAuth

To begin, clone the [GitHub repository for this Quickstart](https://github.com/FusionAuth/fusionauth-quickstart-java-springboot-api) to your local machine:

```bash
git clone https://github.com/FusionAuth/fusionauth-quickstart-java-springboot-api.git
cd fusionauth-quickstart-java-springboot-api
```

Start a local instance of FusionAuth:

```bash
docker compose up -d
```

<Aside type="note">
During the Docker Compose process, the included `kickstart/kickstart.json` file uses the FusionAuth API to configure FusionAuth to a working state, including:

* Generating an API key and configuring CORS
* Creating the ChangeBank application, with a client Id of `e9fdb985-9173-4e01-9d73-ac2d60d1dc8e`
* Creating an admin user (`admin@example.com`, password `password`)
* Creating a teller user (`teller@example.com`, password `password`) with the `teller` role
* Creating a customer user (`customer@example.com`, password `password`) with the `customer` role
</Aside>

Once Docker Compose completes, open a browser window and navigate to the admin UI of your FusionAuth instance at [http://localhost:9011/admin](http://localhost:9011/admin). Log in with the admin credentials above to confirm FusionAuth is running and configured.

<Aside type="note">
If you ever want to reset FusionAuth to this initial state, delete the volumes created by Docker Compose with `docker compose down -v`, then run `docker compose up -d` again.
</Aside>

### Generate the Project

Make a directory for this API:

```shell
mkdir spring-api && cd spring-api
```

<a href='https://start.spring.io/' target='_blank'>Go to the Spring Initializr site</a> and generate your own starter package, adding two dependencies:

- OAuth2 Resource Server
- Spring Web

This example uses:

* Maven
* Spring Boot version `4.0.7`
* Java 17
* A `Group` of `io.fusionauth.quickstart`.
* A `Name` of `FusionAuthQuickstart`.
* An `Artifact` of `springapi`.

If you choose different options, the configuration and code may be different — in particular, the `Group` and `Artifact` values determine the Java package (and therefore the folder path) your own source files will live in, covered in [Add Model Objects](#add-model-objects) below.

Clicking "Generate" on the Spring Initializr site downloads a `springapi.zip` file to your downloads folder, containing the generated project in a top-level `springapi/` directory. Move that zip into `spring-api` and unzip it, then move its contents up a level so the project files sit directly in `spring-api` (matching the `cd spring-api` you did above) rather than in a nested `spring-api/springapi/` directory:

```shell
mv /path/to/downloads/springapi.zip .
unzip springapi.zip
mv springapi/* .
mv springapi/.* .
rm springapi.zip
rmdir springapi
```

<Aside type="note">If you are on Windows substitute `./mvnw` with `.\mvnw.cmd`</Aside>

### Add Security

So far, you haven't done anything with auth, and the controllers are unaware of authentication at all. Now, you'll protect your endpoints based on the `roles` encoded in the JWT you receive from FusionAuth. The decoded payload of a JWT for a `teller` might look like this:

```json
{
  "aud": "e9fdb985-9173-4e01-9d73-ac2d60d1dc8e",
  "exp": 1689289585,
  "iat": 1689285985,
  "iss": "http://localhost:9011",
  "sub": "00000000-0000-0000-0000-111111111111",
  "jti": "ebaa4184-2320-47dd-925b-2e18756c635f",
  "authenticationType": "PASSWORD",
  "email": "teller@example.com",
  "email_verified": true,
  "applicationId": "e9fdb985-9173-4e01-9d73-ac2d60d1dc8e",
  "roles": [
    "teller"
  ],
  "auth_time": 1689285985,
  "tid": "d7d09513-a3f5-401c-9685-34ab6c552453"
}
```

You need to tell Spring how to parse the `roles` out of the claim in the JWT, and for that you need a `Converter`. Add a class `CustomJwtAuthenticationConverter.java`:

<LocalCode src="fusionauth-quickstart-java-springboot-api/complete-application/src/main/java/io/fusionauth/quickstart/springapi/CustomJwtAuthenticationConverter.java" lang="java" />

This class implements the `Converter` interface and takes the `roles` claim and maps it to a `Collection<GrantedAuthority>` Spring will use to authorize the user. We set these as the `authorities` on the AuthenticationToken.

Because you are using a custom converter, you also need to check the audience. Pull out the `aud` claim and validate that at least one of them is in the list of configured audiences for this application.

Now you can add a security configuration. Create a new class `SecurityConfiguration.java`:

<LocalCode src="fusionauth-quickstart-java-springboot-api/complete-application/src/main/java/io/fusionauth/quickstart/springapi/SecurityConfiguration.java" lang="java" />

This class does a lot in a few lines of code. Let's break it down:
- `@Configuration` tells Spring to autowire this configuration class and `@Bean` wires the `SecurityFilterChain`, which will add the rules for handling the security on HTTP requests.
- `OAuth2ResourceServerProperties` is injected into the configuration bean and holds the values you defined in the `application.properties` file.
- The custom `BearerTokenResolver` looks in both the `app.at` cookie and the `Authorization` header for the access token.
- In the `SecurityFilterChain` you do the following:
  1. Get the `audiences` list from the `OAuth2ResourceServerProperties`.
  2. Create a new instance of `CustomJwtAuthenticationConverter` and pass it the `audiences` list.
  3. Tell the filter chain that any requests to `/make-change` must have the `customer` or `teller` role defined in the `GrantedAuthority` list you extract from the JWT.
  4. Tell the filter chain that any requests to `/panic` must have the `teller` role defined in the `GrantedAuthority` list you extract from the JWT.
  5. Set the `oauth2ResourceServer` configuration to use the `CustomJwtAuthenticationConverter` you defined above.
- `JwtDecoder` — this bean tells Spring to decode the JWT in the request using the information from the `issuer` you defined in the properties file.

<Aside type="note">The `NimbusJwtDecoder` will automatically check that the token is not expired based on the `exp` claim.</Aside>

### Call the Protected Endpoints

<Aside type="note">
The code reads the token from either a cookie named `app.at` or the `Authorization` header, so you can use whichever method you prefer in the commands below.

If you use a cookie, make sure you store it in a secure, HttpOnly cookie to avoid exfiltration attacks. See [Storing OAuth Tokens](/articles/oauth/oauth-token-storage) for more information.
</Aside>

Call `/make-change` with a query parameter `total=5.12`, using the token as the `app.at` cookie:

```shell
curl --location 'http://localhost:8080/make-change?total=5.12' \
--cookie 'app.at=<your_token>'
```

Your response should look like this:

```json
{
    "total": 5.12,
    "nickels": 102,
    "pennies": 2
}
```

You're authorized, success! Try the request again without the `Authorization` header, or with an invalid token, and you'll be denied access.

Now call `/panic`, since you're in trouble:

```shell
curl --location --request POST 'http://localhost:8080/panic' \
--cookie 'app.at=<your_token>'
```

This is a `POST`, not a `GET`, because emergency calls should be non-idempotent. Your response should look like this:

```shell
{
    "message": "We've called the police!"
}
```

Nice, help is on the way!

## Next Steps

This is a quick introduction to using FusionAuth to secure an API. To learn more about integrating FusionAuth with your product, see [Get Started](/docs/get-started).

For a production API, consider:

* Using the [Authorization Code grant](/docs/lifecycle/authenticate-users/oauth/#example-authorization-code-grant) instead of the Login API, which keeps credentials within FusionAuth. You can customize the [hosted login pages](/docs/customize/look-and-feel/).
* Using the [Client Credentials grant](/docs/lifecycle/authenticate-users/oauth/#example-client-credentials-grant) to generate a token when calling the API from another service.
* Customizing [token expiration times and policies](/docs/lifecycle/authenticate-users/oauth/#configure-application-oauth-settings) in FusionAuth.
* Making sure you know how to [securely consume a token](/articles/tokens/building-a-secure-jwt#consuming-a-jwt).
* Securing your API [using an API gateway](/docs/extend/examples/api-gateways/) rather than at the framework layer.

FusionAuth has many other features (including [Social Login](/docs/lifecycle/authenticate-users/identity-providers/#social-identity-providers), [Single Sign-On](/docs/lifecycle/authenticate-users/single-sign-on), [Passwordless](/docs/lifecycle/authenticate-users/passwordless/webauthn-passkeys), [Multi-factor Authentication](/docs/lifecycle/authenticate-users/multi-factor-authentication)), most of which are free. Some require additional licensing. For details, see the [pricing page](/pricing).

You can ask general questions in [the FusionAuth forum](/community/forum).

## Troubleshooting

* I get `This site can't be reached localhost refused to connect` when I call the Login API.

  Ensure FusionAuth is running in the Docker container. You should be able to log in as the admin user, `admin@example.com` with a password of `password`, at [http://localhost:9011/admin](http://localhost:9011/admin).

* The `/panic` endpoint doesn't work when I call it.

  Make sure you're making a `POST` call and using a token with the `teller` role.

* I'm getting an error when running `./mvnw spring-boot:run` like this:

  ```
  org.springframework.beans.factory.BeanCreationException: Error creating bean with name 'filterChain' defined in class path resource [io/fusionauth/quickstart/springapi/SecurityConfiguration.class]: Failed to instantiate [org.springframework.security.web.SecurityFilterChain]: Factory method 'filterChain' threw exception with message: Error creating bean with name 'jwtDecoder' defined in class path resource [io/fusionauth/quickstart/springapi/SecurityConfiguration.class]: Failed to instantiate [org.springframework.security.oauth2.jwt.JwtDecoder]: Factory method 'jwtDecoder' threw exception with message: issuer cannot be null
  ```
